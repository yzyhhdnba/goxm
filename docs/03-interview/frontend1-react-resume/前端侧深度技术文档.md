# frontend1 React 前端深度技术文档

这份文档不是给简历直接摘抄的，而是给面试深挖时使用。重点是把“为什么这么做”讲清楚。

---

## 1. 为什么要采用并行 React 前端，而不是直接替换 Vue

项目背景里已经存在一套 Vue 前端，它本身承载的是业务事实和页面语义。如果直接替换主前端，会有两个风险：

- 一是容易把“技术重写”混成“业务改造”，导致范围失控
- 二是没有一个稳定对照物，不利于验证 React 版到底是不是把真实链路做通了

所以这里采取的是并行策略：

- `frontend/` 继续作为现有主前端
- `frontend1/` 作为独立 React 前端工程

这样做的价值：

- 能保留原有业务流程作为参照
- 能逐步验证 React 页面和后端接口是否一致
- 能把 React 重构项目讲成“工程迁移与并行交付”而不是“从零搭一个 demo”

## 2. 状态管理为什么选择 Redux Toolkit + Hooks 混合方案

这个项目里的状态分三类：

### 2.1 全局共享状态

适合放 Redux Toolkit：

- 登录态
- 当前用户信息
- 角色信息
- 全局权限判断

原因：

- 这些状态跨页面共享
- 页面切换时需要稳定读取
- 路由守卫和头部导航都会依赖

### 2.2 页面内临时状态

适合用 `useState`：

- 表单输入
- 当前 tab
- 弹窗开关
- 局部 loading
- 当前分页页码对应的本地展示状态

原因：

- 生命周期跟随当前页面
- 没有跨页面共享价值
- 放全局反而会增加维护成本

### 2.3 URL 驱动状态

这部分不是放 Redux，而是直接放到路由里：

- 搜索关键词
- 当前页码
- 当前 tab
- 来源页 from

原因：

- 这些状态本质上是“页面位置”的一部分
- 只有放到 URL 里，刷新、复制链接、浏览器返回才有一致行为

## 3. 统一请求层为什么重要

项目里没有让页面直接写裸请求，而是统一走 API 模块。

这样做的作用有三点：

### 3.1 统一 token 和鉴权失效逻辑

- 请求前统一注入 token
- 401 统一清理会话
- 路由守卫只关心当前登录态是否有效

### 3.2 统一数据返回结构

后端返回是统一包裹结构，前端请求层先做解包，再交给页面消费。  
这样页面拿到的就已经是业务数据，而不是每个页面自己判断 `code / message / data`。

### 3.3 降低业务模块耦合

页面调用的是 `SearchAPI / VideoAPI / UserAPI / UploadAPI / AdminAPI / NoticeAPI` 这些按业务域划分的模块，后面如果接口路径或参数有变化，主要影响 API 层，不会把修改扩散到所有组件。

## 4. 评论树为什么是代表性实现

评论区是这个 React 前端里最能体现“复杂交互”的模块之一。

它的难点不只是把数据展示出来，而是：

- 一级评论和二级回复是两层结构
- 回复列表需要展开和收起
- 回复既可以回复楼主，也可以回复某条回复
- 点赞和回复发布后，最好局部更新，不要整页重载
- 回复数较多时还需要分页

React 版的处理思路是：

- 一级评论单独做成 `CommentThread`
- 回复列表单独做成 `ReplyThread`
- 交互尽量局部更新当前节点
- 点赞操作使用乐观更新，失败时回滚
- 回复分页只作用在当前评论节点，不影响整个评论区

这个点适合回答“你怎么拆组件”“你怎么处理复杂交互”“你怎么平衡局部状态和全局状态”这类问题。

## 5. 为什么强调来源页透传和条件保留

很多同学做页面跳转时，只关注“能跳过去”，但面试官更容易被打动的是“跳回来还能保持原状态”。

这个项目里比较典型的链路有：

- 搜索结果 -> 用户主页 -> 视频详情 -> 返回
- 历史记录 -> 视频详情 -> 作者主页 -> 返回
- 用户主页 -> 视频详情 -> 作者主页卡片 -> 回原用户主页状态

如果不处理来源页和条件保留，用户一返回就可能丢掉：

- 搜索关键词
- 搜索结果类型
- 页码
- 用户主页 tab

React 版的处理方式是：

- 把关键词、tab、page 放到 URL
- 再增加 `from` 记录来源路径
- 详情页和用户主页继续向下透传 `from`

这样做的价值是：

- 浏览器返回行为更自然
- 页面状态更稳定
- 也更适合面试里展示你对真实用户体验的关注

## 6. 投稿和审核为什么适合当成一条链来讲

如果只讲投稿页面，容易显得像“普通表单上传”；  
如果只讲审核页面，容易显得像“后台列表操作”。

把它们放在一起讲，才更像真实业务：

1. 创作者填写元数据
2. 上传原生视频
3. 上传封面或截取当前帧做封面
4. 稿件进入待审状态
5. 管理员审核通过或驳回
6. 用户在通知页和通知气泡中看到审核结果

这条链路能体现：

- 前端表单和文件上传处理能力
- 复杂页面拆分能力
- 用户侧与管理侧联动能力
- 业务闭环思维

## 7. 如果面试官继续深挖，可以怎么回答“后续优化方向”

推荐按下面三类说：

### 7.1 体验层优化

- 上传页做时间轴选帧器
- 通知做轮询或实时推送
- 分页控件和交互风格统一

### 7.2 工程层优化

- 增加更细的类型约束
- 给复杂页面补单元测试和集成测试
- 抽离更多跨业务通用组件

### 7.3 业务层优化

- 创作者稿件编辑体验继续增强
- 审核后台补筛选和搜索
- 通知与上传结果做更多联动提示

这样回答的好处是：

- 既能说明你知道当前项目还不完美
- 又能体现你对后续迭代路线有思考
