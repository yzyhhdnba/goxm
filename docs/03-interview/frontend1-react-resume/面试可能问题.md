# frontend1 React 前端面试可能问题

## 1. 为什么原来已经有 Vue 前端，你还要新建一个 React 前端？

建议回答：

> 我不是为了推翻原有前端，而是想在不破坏现有业务的前提下，验证自己能不能把一套复杂业务用 React 技术栈重新搭起来。原来的 Vue 前端相当于业务参考，而 `frontend1/` 更像一个并行重构实验场。这样既保留了已有业务语义，也能体现 React 工程化和前后端联调能力。

---

## 2. 这个项目里你为什么选 Redux Toolkit，而不是只用 useState？

建议回答：

> 因为这个项目里既有页面级局部状态，也有跨页面共享状态。像登录态、当前用户、权限信息、来源页链路这些更适合放全局；而表单输入、弹窗开关、局部加载状态更适合留在组件内。Redux Toolkit 主要解决共享状态组织问题，`useState` 解决页面内临时交互，两者是配合关系。

---

## 3. 为什么还要自己封装 Axios？

建议回答：

> 因为如果不封装，请求层逻辑会散落在各个页面里。封装之后可以统一处理 token 注入、响应解包、错误收口和 401 失效逻辑，页面只关心业务数据本身。这样联调时也更稳定，后面加新的模块也更快。

---

## 4. 你觉得这个项目最能体现前端能力的部分是什么？

建议回答：

> 我觉得不是单个页面，而是多页面之间的链路控制。比如搜索页进入用户主页，再进入视频详情，再返回上一页，关键词、页码和 tab 都要保留下来。这个过程不难写，但很考验前端对路由状态、页面来源和用户体验的处理。

---

## 5. 评论树这块你做了什么？

建议回答：

> 我把一级评论和二级回复拆成评论树组件来做。一级评论支持点赞和展开回复，回复列表支持分页、发布回复和回复点赞，整个过程是局部刷新，不会整页重载。这样既能体现组件拆分能力，也能说明我考虑了真实互动场景下的体验。

---

## 6. 投稿页最有意思的点是什么？

建议回答：

> 投稿不是只做个表单提交，我把它拆成了元数据、视频文件、封面文件三部分。封面除了支持手动上传，还支持从视频预览当前帧直接截取，这个是基于浏览器原生 `video + canvas` 做的。这个点比较适合展示我不只会调接口，也会结合浏览器能力设计交互。

---

## 7. 管理后台和通知体系你怎么讲？

建议回答：

> 我把管理后台和通知放在一起讲，因为它们在业务上是一条链。管理员在审核页里通过或驳回稿件，创作者会在通知页和头部通知气泡里看到审核结果。这样面试官更容易感受到这不是两个孤立页面，而是一条完整业务闭环。

---

## 8. 这个项目还有哪些不足？

建议回答：

> 第一，上传页现在只是“当前帧截封面”，还不是完整的视频时间轴选帧器；第二，通知气泡目前是首次拉取，不是实时更新；第三，很多页面已经打通主链路，但 UI 细节和分页样式还可以继续统一。我的思路一直是先把真实业务闭环做通，再做体验层面的打磨。
