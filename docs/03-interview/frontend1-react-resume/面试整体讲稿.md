# frontend1 React 前端面试整体讲稿

## 1. 1 分钟版本

> 我在这个项目里做的是一个并行 React 前端重构。原项目主前端是 Vue，但为了练 React 工程化和业务迁移能力，我新建了 `frontend1/`，用 React + TypeScript + Vite + Redux Toolkit 把视频平台核心页面重新实现了一遍。  
> 我不是只做静态页面，而是把首页推荐、搜索、视频详情、评论回复、个人中心、历史记录、创作者投稿、审核后台和系统通知都接到了真实的 Go 后端接口上。  
> 这个项目里我觉得最能体现前端能力的点主要有三个：第一是统一请求层和路由状态管理；第二是评论树、搜索跳转和来源页保留这类真实交互细节；第三是把用户侧、创作者侧、管理侧三条链路都补成了完整业务闭环。

## 2. 3 分钟版本

可以按下面这个顺序讲：

1. 为什么要做并行 React 前端
2. 技术栈和工程结构
3. 我重点做了哪些业务链路
4. 这个项目里最能体现前端能力的点是什么

参考话术：

> 这个项目原来已经有一套 Vue 前端，我没有直接替换它，而是在根目录新建了 `frontend1/`，独立用 React 技术栈重构核心页面。这样做的价值是，一方面保留原有业务流程作为参考，另一方面也能验证我是不是能把一套复杂业务用 React 工程化方式重新搭起来。  
> 技术上我用的是 React、TypeScript、Vite、Tailwind CSS、Redux Toolkit、React Router 和 Axios。状态管理上，全局登录态和共享状态放 Redux，页面内的临时交互状态用 `useState` 管。请求层做了统一封装，集中处理 token、错误和返回结构。  
> 业务上我重点做了首页推荐、搜索、视频详情、评论回复、用户主页、个人中心、历史记录、创作者投稿、管理员审核和系统通知。也就是说，这不是单页面 demo，而是覆盖了用户侧、创作者侧和管理侧三条主链路。  
> 我觉得这里最值得讲的点，一是评论树和回复交互，二是搜索页、用户主页、视频详情之间的来源页透传和分页条件保留，三是投稿和审核这一整条复杂业务链路的前端闭环。

## 3. 5 分钟版本

### 3.1 项目定位

- 这是一个“基于真实后端接口的 React 前端重构项目”
- 重点不在于炫技，而在于把复杂业务页面用 React 工程化方式稳定落地
- 适合展示组件拆分、状态管理、接口联调和页面链路控制能力

### 3.2 技术方案

- 用 Vite 重新搭 React 工程，保证开发和构建链路独立
- 用 TypeScript 做接口和状态约束
- 用 Redux Toolkit 管全局登录态、用户信息和共享状态
- 用 React Router 管页面跳转和来源页保留
- 用 Axios 封装统一请求层

### 3.3 代表性业务链路

#### 链路一：搜索 -> 用户主页 -> 视频详情 -> 返回

- 搜索页支持视频和用户两种结果
- 通过 URL 保存关键词、分页、tab 和来源页
- 用户主页和详情页继续传递来源路径
- 用户返回时不会丢失原来的上下文状态

#### 链路二：视频详情 -> 评论树 -> 回复树

- 详情页拉视频主信息、评论列表和推荐内容
- 一级评论支持点赞、展开回复
- 回复树支持分页、继续回复和回复点赞
- 整体是局部刷新，不需要整页重载

#### 链路三：创作者投稿 -> 管理员审核 -> 通知提醒

- 创作者先填写元数据，再上传原生视频和封面
- 封面支持手动上传，也支持截取视频当前帧
- 管理员在审核后台执行通过或驳回
- 用户右上角通知气泡和通知页会显示审核结果

### 3.4 我认为这个项目的亮点

- 不是只实现页面，而是把三类角色链路串成闭环
- React 工程结构比较清晰，业务域拆分明确
- 对真实交互细节处理比较完整，比如来源页透传、分页保留、局部更新和状态回滚

## 4. 结束时的收口话术

> 如果继续迭代，我会优先补三个方向：第一是把上传页做成更完整的视频选帧器；第二是把通知气泡升级成轮询或实时推送；第三是继续统一分页控件和交互风格，让整个 React 前端在体验上更完整。
